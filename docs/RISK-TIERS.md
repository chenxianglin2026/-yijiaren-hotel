# 风险分级标准 (Risk Tiers)

> AI 执行任何操作前必须判定 Tier 等级。Tier 2+ 必须在报告中注明风险和回滚方案。

---

## Tier 0 - 无风险 (自主执行)

**定义**: 纯本地操作，不影响任何运行中的服务。

| 操作类型 | 示例 |
|---------|------|
| 文件读写 | 读写代码、文档、测试 |
| 代码修改 | patch、write_file 编辑项目文件 |
| 本地测试 | pytest、node --check |
| Git 操作 | git add、commit、status、log |
| 搜索/查找 | search_files、read_file |
| 文档生成 | 创建/更新 .md 文件 |

**执行规则**: 直接执行，无需报告。

---

## Tier 1 - 低风险 (自主执行，批量报告)

**定义**: 涉及远程但无破坏性的操作。

| 操作类型 | 示例 |
|---------|------|
| Git push | 普通 push (非 force) |
| VPS 查询 | ssh 执行只读命令 (docker ps, ls, cat, curl get) |
| API 测试 | curl GET/POST 测试端点 |
| 依赖安装 | pip install, npm install (本地) |
| 构建/编译 | docker build (本地) |

**执行规则**: 直接执行，在轮次报告中简要汇总。

---

## Tier 2 - 中风险 (执行前报告中注明)

**定义**: 影响运行中服务的操作，但有明确回滚方案。

| 操作类型 | 示例 |
|---------|------|
| 容器重启 | docker restart |
| Nginx reload | nginx -s reload (宿主机) |
| 数据库迁移 | ALTER TABLE, schema 变更 |
| 端口开放 | 新增端口映射 |
| 大文件删除 | git filter-branch |
| 容器重建 | docker compose up -d --build |

**执行规则**:
1. 报告中注明：影响范围 + 预计影响时长 + 回滚方案
2. 执行后验证
3. 记录到 daily 复盘

**回滚标准**: 必须有明确的逆向操作 (如: 备份文件已存于 /tmp/xxx.bak)

---

## Tier 3 - 高风险 (需人工审批)

**定义**: 不可逆或影响生产安全的操作。

| 操作类型 | 示例 |
|---------|------|
| 生产数据库变更 | DROP TABLE, DELETE FROM |
| iptables 修改 | 防火墙规则变更 |
| DNS 变更 | 域名解析修改 |
| 密钥/证书变更 | SSL 证书替换、AppSecret 重置 |
| Force push | git push --force |
| 删除生产资源 | docker rm, 删容器/镜像 |
| 微信发布 | 小程序提交审核/发布 |
| 第三方 API 密钥 | 生产环境密钥配置 |

**执行规则**:
1. 必须向陈总报告：操作内容 + 影响范围 + 风险 + 回滚方案
2. 等待确认后方可执行
3. 执行后立即验证
4. 强制记录到 daily 复盘 + lessons

---

## 快速判定流程

```
1. 这个操作会影响到运行中的服务吗？
   NO  → Tier 0
   YES → 继续

2. 操作可逆吗？有回滚方案吗？
   YES → Tier 1 或 Tier 2
   NO  → Tier 3 (需审批)

3. 出错了多久能恢复？
   秒级 → Tier 1
   分钟级 → Tier 2
   不可恢复 → Tier 3
```
